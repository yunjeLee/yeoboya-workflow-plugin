# module-claude-template 모듈

호출자(`skills/harness-module/SKILL.md`) 가 Read tool 로 읽고 그대로 따른다.
이 모듈은 모듈별 `CLAUDE.md` 의 양식, 자동 추론 가이드 (Q1/Q4 및 Q2/Q3/Q5), 50 줄 압축 규칙, 그리고 `docs/MODULE_MAP.md` 인덱스 양식을 한 곳에 정의한다.

전제: 호출자는 `shared/module-detect.md` 를 먼저 실행해 `modules` (각 항목은 `{ name, path, file_count, has_namespace, depends_on, dependents, has_readme, recommend }`) 를 들고 진입한다. 본 모듈은 그 결과를 기반으로 모듈별 CLAUDE.md 본문을 생성한다.

---

## 모듈 CLAUDE.md 양식 (≤50 줄)

각 모듈의 `{path}/CLAUDE.md` 는 아래 양식으로 작성한다. 5 개 H2 섹션은 누락 없이 모두 포함한다.

````markdown
# {모듈명}

## 역할 (Q1)
{한 줄 요약. 자동 추론 → 사용자 검토}

## 표준 작업 흐름 (Q2)
{이 모듈에서 반복되는 추가/수정/삭제 절차 1~3 줄. 또는 "해당 없음"}

## 절대 하면 안 되는 것 (Q3)
- {1 줄씩. 또는 "해당 없음"}

## 의존성 (Q4)
- 의존하는 모듈: {빌드 그래프 자동}
- 이 모듈에 의존하는 모듈: {빌드 그래프 자동}

## 명시되지 않은 규칙 (Q5)
- {1 줄씩. 또는 "해당 없음"}
````

- `{모듈명}` 에는 `module-detect.md` 의 `name` 을 그대로 쓴다 (Android 콜론 표기 `:feature:home`, iOS 타겟명 `FeatureHome` 등 — 표기 규칙은 module-detect.md 가 결정한다).
- 빈 섹션이라도 헤더는 남기고 `해당 없음` 으로 채운다 — "잊고 안 적은 것" 과 "의도해서 비운 것" 을 구분하기 위함.

## 자동 추론 가이드 (Q1 / Q4)

자동으로 채울 수 있는 항목은 호출자가 미리 채워서 사용자에게는 검토만 시킨다.

### Q1 (역할) 자동 추론

다음 신호를 조합해 한 줄 초안을 만든다:

1. 모듈 디렉토리/패키지명: `module-detect.md` 의 `name` (콜론 표기) 과 `path` 마지막 세그먼트.
2. README 첫 단락: `has_readme == true` 면 `{path}/README.md` 를 Read 해서 첫 번째 비어 있지 않은 단락의 첫 문장.
3. public API 시그니처:
   - Android: `{path}/**/*.kt` 중 `public interface` / `public class` / `interface` / `class` (Kotlin 의 기본 가시성이 public 이므로 명시 modifier 가 없는 top-level 선언도 포함) 의 이름과 public 메서드 시그니처 상위 5 개.
   - iOS: `{path}/**/*.swift` 중 `public func` / `open class` / `public struct` / `public protocol` 시그니처 상위 5 개.

위 셋을 조합해 "이 모듈은 ~ 를 담당한다" 형태로 한 줄 초안을 만든다. 사용자는 검토 단계에서 수정하거나 그대로 둘 수 있다.

### Q4 (의존성) 자동 추론

`shared/module-detect.md` 의 `## 모듈 메타정보 수집` 결과를 그대로 옮긴다.

- "의존하는 모듈" = `module-detect.md` 에서 수집한 `depends_on` 배열을 그대로 사용 (정방향 그래프).
- "이 모듈에 의존하는 모듈" = `module-detect.md` 에서 수집한 `dependents` 배열을 그대로 사용 (역방향 그래프).
- 어느 쪽이든 비면 `(없음)` 표기. 사용자는 검증만 한다.

## 자동 추론 가이드 (Q2 / Q3 / Q5)

Q1/Q4 와 별도 절차다. Q2/Q3/Q5 는 코드 + 기존 하네스 문서를 분석해 초안을 만들고 사용자에게는 한 번만 검토를 받는다.

### Q2 (표준 작업 흐름) 자동 초안

모듈 내 클래스 호출 체인을 분석한다.

1. 모듈 내 ViewModel / UseCase / Repository / DataSource / Api 류 클래스 탐색.
   - Android: 클래스명 suffix `*ViewModel`, `*UseCase`, `*Repository`, `*DataSource`, `*Api`.
   - iOS: `*ViewModel`, `*UseCase`, `*Repository`, `*Service`, `*Client`.
2. 가장 빈도 높은 호출 체인 1 개를 추출 (예: `LoginViewModel` → `LoginUseCase` → `AuthRepository`).
3. Intent / State / SideEffect sealed class 가 있으면 MVI 흐름임을 표기.
4. 초안 형식 — 단계 3 의 MVI 감지 결과로 분기:
   - MVI 흐름 감지: "Intent 추가 → ViewModel.handleIntent 분기 → UiState 필드 → reducer 갱신"
   - MVI 미감지: "ViewModel → UseCase → Repository 순으로 호출, suspend 만 사용"

호출 체인 추출 실패 (모듈에 위 패턴 클래스 없음) 시 초안은 `(자동 추출 실패 — 사용자가 [e] 로 직접 입력 또는 [s] 해당 없음 권장)`.

### Q3 (절대 하면 안 되는 것) 자동 초안

기존 하네스 문서 + 코드 위반 사례를 결합한다.

1. `docs/CONVENTIONS.md` Read → 모듈 책임에 맞는 금기 항목 1~3 개 추출 (예: feature 모듈이면 "Composable 안에서 Repository 호출 금지").
2. `docs/ADR.md` Read → ADR 의 "안 하기로 한 것" 결정사항 1~2 개 추출.
3. 모듈 내 코드에서 단계 1/2 의 금기 키워드를 Grep 으로 텍스트 매칭. 매칭 시 초안에 `(잠재 위반: {파일:줄})` 첨부. 매칭 실패는 단계 1/2 만으로 초안 작성.
4. 초안 형식: 1 줄당 1 항목, 최대 5 줄.

위 1~3 모두 결과 없으면 초안은 `(자동 추출 실패 — 사용자가 [e] 로 직접 입력 또는 [s] 해당 없음 권장)`.

### Q5 (명시되지 않은 규칙) 자동 초안

모듈 내 통계 패턴을 추출한다.

1. 클래스 네이밍 suffix 빈도 (상위 3 개): "ViewModel 은 *ViewModel 로 끝낸다" 같은 규칙 도출.
2. 패키지 / 디렉토리 분할 (presentation/domain/data 또는 ui/data/domain 등 레이어 분포).
3. 에러 처리 방식: sealed class `*Error` / `Result<T>` / `Either` / `try-catch` 분포 중 가장 많은 것.
4. 초안 형식: 1 줄당 1 항목, 최대 5 줄.

통계 표본이 5 개 미만 (모듈에 클래스가 너무 적음) 이면 초안은 `(자동 추출 실패 — 사용자가 [e] 로 직접 입력 또는 [s] 해당 없음 권장)`.

### 검토 프롬프트 (모듈당 1회)

Q2 / Q3 / Q5 초안을 한꺼번에 출력 후 한 번만 묻는다.

```
{모듈명} 자동 초안 (Q2/Q3/Q5):

## 표준 작업 흐름 (Q2)
{Q2 초안}

## 절대 하면 안 되는 것 (Q3)
{Q3 초안}

## 명시되지 않은 규칙 (Q5)
{Q5 초안}

[Y] 그대로 수락 / [e] 일부 수정 / [s] 해당 없음 적용 (한 항목 또는 전체)
```

- `Y` (또는 빈 입력): 초안 그대로 사용. 다음 단계로.
- `e`: 어떤 항목을 어떻게 바꿀지 자유 입력 받아 1 회 반영 후 다시 [Y/e/s] 묻기 (최대 3 회 반복).
- `s`: 항목 지정 입력. 형식:
  - `s` (단독): "어떤 항목? (Q2 / Q3 / Q5 / 전체)" 를 1 회 추가로 묻는다.
  - `s Q2` / `s Q3,Q5` / `s 전체`: 즉시 적용. 지정된 항목만 `해당 없음` 으로 채우고 나머지는 초안 유지.

> **의도된 공백 vs 잊은 공백** 정책은 유지된다. 단 표현이 바뀐다 — 이전에는 빈 답변을 막았지만, 지금은 자동 초안이 모든 항목에 무언가를 채우므로 사용자가 명시적으로 `[s]` 를 입력해야만 `해당 없음` 이 된다. 빈 입력 = `[Y]` (자동 수락).

질문 round-trip 은 모듈당 1 회. `[e]` 로 수정이 발생하면 위 3 회 한도가 적용된다.

## 50 줄 한도 압축 규칙

CLAUDE.md 가 길어지면 모델이 매번 다 읽지 못한다. 모듈별로 50 줄 한도를 강제한다.

1. 작성 직후 줄 수를 측정한다: `wc -l {path}/CLAUDE.md`.
2. 50 줄 이하면 그대로 저장. 종료.
3. 51 줄 이상이면 자동 압축에 들어간다:
   - 각 H2 블록 안에서 중복/장황한 표현을 1 줄씩 줄여 50 줄 이하로 맞춘다.
   - 5 개 H2 (역할 / 표준 작업 흐름 / 절대 하면 안 되는 것 / 의존성 / 명시되지 않은 규칙) 와 각 섹션의 핵심 정보 (역할 한 줄, 작업 흐름 핵심 단계, 금기 항목, 의존성 양방향, 컨벤션) 는 절대 삭제하지 않는다.
   - 압축 후본을 사용자에게 출력하고 다음과 같이 묻는다:
     ```
     [Y/n] 이 압축본으로 저장할까요?
     ```
   - `Y` (또는 빈 입력) → 저장. `N` → 저장하지 않고 사용자가 직접 줄여서 다시 입력하도록 안내.
4. **정보 손실 방지**: 압축이 발생할 때마다 원문(압축 전) 을 호출자가 사용 중인 progress 파일 (예: `docs/superpowers/progress/harness-module.md`) 의 모듈별 `### 백업 — {모듈명}` 섹션으로 남긴다. 사후 복원 가능해야 한다.

## docs/MODULE_MAP.md 인덱스 양식

전체 모듈을 한눈에 보기 위한 인덱스 파일은 다음 양식으로 작성한다.

````markdown
# 모듈 인덱스

> `/harness-module` 으로 생성됨. 모듈 추가/삭제 시 다시 실행.

| 모듈 | 역할 (Q1 1 줄) | 의존하는 모듈 | 이 모듈에 의존하는 모듈 |
|------|----------------|---------------|--------------------------|
| [`{모듈명1}`]({모듈경로1}/CLAUDE.md) | {Q1 첫 문장} | `{a}`, `{b}` | `{c}` |
| [`{모듈명2}`]({모듈경로2}/CLAUDE.md) | {Q1 첫 문장} | (없음) | `{a}`, `{b}` |
````

작성 규칙:

- "역할" 셀에는 Q1 본문의 **첫 문장만** 적는다 — 마침표(`.`) 또는 줄바꿈 직전까지. 길면 잘라 쓴다.
- "의존" 셀에 항목이 없으면 `(없음)` 표기 (빈 셀 금지 — 의도와 누락 구분).
- 모듈명 셀은 클릭 가능한 링크로 — `{모듈경로}/CLAUDE.md` 로 1 hop 으로 가게 한다.
- 행 정렬은 `module-detect.md` 가 반환한 모듈 순서를 그대로 유지한다 (gradle / Tuist / SwiftPM 의 선언 순서 보존).
